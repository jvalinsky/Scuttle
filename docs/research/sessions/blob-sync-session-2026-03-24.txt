    This Session: Blob Sync Over Room Tunnel

    What We Built

    Added testBlobSyncThroughRoom — a new integration test that syncs a blob between two SSBRoomClient instances via a Docker room relay.

    ---
    Bugs Found & Fixed

    1. "Duplicate tunnel" collision kills the peer session (SSBRoomClient.m)
    - Problem: When both clients auto-connect to each other simultaneously, the tiebreak sent an error frame {name: Error, message: "duplicate tunnel"}. go-muxrpc treats error frames as fatal → calls
    w.CloseWithError() → terminates the entire peer session, not just the stream.
    - Fix: Changed to send @YES with EndErr|Stream (clean stream end) so go-muxrpc closes only that stream gracefully.

    2. Source stream end signal never reached the consumer (SSBMuxRPCSession.m)
    - Problem: handleIncomingMessage: had a gap: when a stream ends with @YES (the MuxRPC convention for clean stream termination), or with an empty body, it silently removed the pending callback without
    invoking it. So fetchBlob:session:completion: accumulated data but was never told "stream done" — completion block never fired.
    - Fix: Added else { callback(nil, nil); } for the clean-stream-end case, so all source stream consumers get a proper end signal.

    3. handleBlobGetRequest: sent wrong stream terminator (SSBRoomClient.m)
    - Problem: Used sendData:nil forRequest:isEnd:YES which produces a no-type-flag empty frame — technically malformed per MuxRPC spec.
    - Fix: Changed to sendData:@YES forRequest:isEnd:YES — proper MuxRPC source stream terminator (JSON true with Stream+EndErr flags).

    4. Test timeouts too tight (SSBDockerRoomIntegrationTests.m)
    - Legacy tunnel.endpoints discovery takes ~34s due to polling interval, but bSeesA timeout was 30s.
    - Fix: Increased bSeesA wait to 60s; tightened blobFetched to 15s (since the actual fetch is now near-instant once the tunnel is up).

    ---
    Key Insight About MuxRPC Stream Semantics

    @YES with EndErr|Stream is the SSB/MuxRPC convention for "clean stream end" — it is not data, it is a sentinel. Consumers expect (nil, nil) to mean end-of-stream. The session layer was swallowing this
    sentinel without notifying consumers, breaking any source-stream protocol built on top (blobs, potentially others).

     This Session: Blob Sync Over Room Tunnel

    What we added

    testBlobSyncThroughRoom — a new integration test where Client B fetches a blob from Client A through the Docker room tunnel.

    ---
    Bugs found and fixed

    1. blobs.get was unhandled in tunnel sessions
    - Any non-EBT RPC request arriving on a tunnel session was silently ignored ("Ignoring non-EBT RPC request")
    - Added handleBlobGetRequest:requestID:session: dispatch in handleEBTMessage: and the implementation that reads from self.blobStore and streams data back

    2. "Duplicate tunnel" error frame killed the entire peer session
    - When two clients simultaneously initiated tunnel.connect to each other, the tiebreak rejection sent @{name:Error, message:duplicate tunnel} as an error frame
    - go-muxrpc treats error frames by calling w.CloseWithError() — terminating the entire peer session, not just the stream
    - Fixed: changed rejection to a clean stream end (@YES with EndErr|Stream) so go-muxrpc closes only that stream

    3. MuxRPC stream end never called the consumer callback
    - handleIncomingMessage: in SSBMuxRPCSession.m silently dropped the @YES end-of-stream sentinel without invoking the pending callback
    - fetchBlob:session:completion: waited for (nil, nil) to know all chunks had arrived — it never came
    - Fixed: added else { callback(nil, nil); } for the clean-stream-end case so all source-stream consumers get a proper end signal

    4. Test timeouts too tight
    - Legacy tunnel.endpoints polling takes ~34s, but bSeesA timeout was 30s
    - Fixed: 30s → 60s for endpoint discovery; blob fetch timeout tightened to 15s (now completes in <1s after tunnel ready)

    ---
    Key architectural insight

    The tunnel's receiveRequestBlock is handleEBTMessage: for all MuxRPC traffic over a peer tunnel — not just EBT frames. Any new peer protocol (blobs, invite redemption, etc.) needs a dispatch branch
    there.
